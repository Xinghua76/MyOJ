# OJ 项目系统体检与整改建议（答辩版）

更新时间：2026-02-21

## 1. 结论概览
- 当前项目主流程可编译运行（后端与沙箱 `mvn -DskipTests compile` 通过）。
- 系统已具备完整业务链路，但存在若干高风险细节，主要集中在：
  - 配置与密钥安全
  - 鉴权边界
  - CORS 策略
  - 沙箱调用安全与环境可配置性
- 若用于答辩演示，建议优先修复 P0/P1 问题，避免被追问时失分。

## 2. 问题分级清单（按严重程度）

### P0（必须先修）
1. 明文密钥/口令出现在配置文件
- 证据：
  - `yuoj-backend-master/src/main/resources/application.yml:22`
  - `yuoj-backend-master/src/main/resources/application.yml:29`
  - `yuoj-backend-master/src/main/resources/application.yml:53`
  - `yuoj-backend-master/src/main/resources/application.yml:83`
  - `yuoj-backend-master/src/main/resources/application-prod.yml:13`
- 风险：凭据泄露、答辩安全性扣分、上线不可接受。
- 建议：全部改为环境变量注入，仓库仅保留 `application.example.yml`。

2. 沙箱鉴权与地址硬编码
- 证据：
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/judge/codesandbox/impl/RemoteCodeSandbox.java:19`
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/judge/codesandbox/impl/RemoteCodeSandbox.java:24`
  - `yuoj-code-sandbox-master/src/main/java/com/yupi/yuojcodesandbox/controller/MainController.java:21`
- 风险：密钥固定、环境不可切换、生产脆弱。
- 建议：`auth` 密钥与 URL 全部走配置项，按环境区分。

3. CORS 过宽且允许凭据
- 证据：
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/config/CorsConfig.java:21`
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/config/CorsConfig.java:23`
- 风险：跨站携带 Cookie 风险高。
- 建议：仅放行白名单域名（开发、测试、生产分别配置）。

4. 封禁用户权限校验存在漏洞
- 证据：
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/aop/AuthInterceptor.java:55`
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/service/impl/UserServiceImpl.java:178`
- 风险：被封禁用户在已登录会话下仍可能访问部分接口。
- 建议：在统一登录态校验处直接拒绝 `ban`，并清理会话。

### P1（答辩前建议修）
1. 报名分页接口缺后端鉴权
- 证据：
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/controller/ContestSignupController.java:113`
- 风险：报名数据可被未授权枚举。
- 建议：至少要求登录；更稳妥是仅管理员可查全量。

2. 密码加密算法偏弱（固定盐 MD5）
- 证据：
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/service/impl/UserServiceImpl.java:36`
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/service/impl/UserServiceImpl.java:65`
- 风险：撞库与离线破解风险高。
- 建议：升级 BCrypt/Argon2，并提供兼容迁移逻辑。

3. 前端 OpenAPI 基址硬编码 localhost
- 证据：
  - `yuoj-frontend-master/generated/core/OpenAPI.ts:23`
- 风险：环境切换易失效，演示机器常见“接口全挂”。
- 建议：通过 `.env.*` 注入 API Base URL。

### P2（可作为“后续演进”）
1. 沙箱控制器默认走 Native 实现，和“容器隔离”叙述不一致
- 证据：
  - `yuoj-code-sandbox-master/src/main/java/com/yupi/yuojcodesandbox/controller/MainController.java:24`
- 建议：用配置开关明确 `native/docker`，并在答辩中解释差异。

2. Docker 沙箱镜像初始化逻辑可优化
- 证据：
  - `yuoj-code-sandbox-master/src/main/java/com/yupi/yuojcodesandbox/JavaDockerCodeSandbox.java:31`
  - `yuoj-code-sandbox-master/src/main/java/com/yupi/yuojcodesandbox/JavaDockerCodeSandbox.java:62`
- 建议：避免每次判题路径重复触发拉镜像。

3. 部分页接口缺统一 pageSize 上限
- 证据：
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/controller/ContestController.java:102`
  - `yuoj-backend-master/src/main/java/com/yupi/yuoj/controller/ContestSignupController.java:115`
- 建议：统一限制（如最大 50）并做参数兜底。

## 3. 构建与可运行性检查
- 后端：`mvn -DskipTests compile` 通过。
- 沙箱：`mvn -DskipTests compile` 通过。
- 前端：当前环境构建受权限策略影响（`spawn EPERM`），非业务逻辑报错，但会影响本机演示。

## 4. 答辩表达建议（可直接复述）
1. “核心工程价值不在页面数量，而在判题链路的可用性和可恢复性：提交异步化、失败兜底、权限隔离、日志审计。”
2. “我们已识别出配置与边界安全问题，并按 P0/P1/P2 形成整改计划，优先保证可演示、可上线的底线。”
3. “系统后续优化方向为：密钥治理、密码升级、跨域白名单、沙箱配置化与性能优化。”

## 5. 整改排期（建议）
- 2 小时内：处理 P0 中配置外置、CORS 白名单、沙箱 URL/密钥配置化。
- 半天内：补齐报名分页鉴权、封禁用户统一拦截。
- 1 天内：密码升级方案（BCrypt + 兼容迁移）与前端环境变量改造。

## 6. 交付物建议（便于老师验收）
- 一份脱敏后的 `application.example.yml`。
- 一份“安全与稳定性整改记录”表（问题、责任人、状态、完成日期）。
- 一段 3 分钟演示脚本：提交判题 -> 队列消费 -> 结果回写 -> 管理日志可追踪。
